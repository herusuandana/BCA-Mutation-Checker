# .gitignore Documentation

This document explains what files are ignored by Git and why.

## Critical Files (NEVER COMMIT!)

### config/config.json

**Why**: Contains sensitive credentials (username, password, proxy credentials)
**Risk**: Exposing credentials to public repositories
**Alternative**: Use `config/config.example.json` as template

### .env files

**Why**: Contains environment variables and secrets
**Risk**: Exposing API keys, tokens, passwords
**Alternative**: Document required variables in README

## Build Artifacts

### node_modules/

**Why**: Large directory with dependencies (can be 100MB+)
**Risk**: Bloating repository size
**Alternative**: Use `package.json` and `npm install`

### dist/, build/

**Why**: Compiled/transpiled code (generated from source)
**Risk**: Unnecessary duplication, merge conflicts
**Alternative**: Build from source using `npm run build`

### \*.tsbuildinfo

**Why**: TypeScript incremental build cache
**Risk**: Not portable across machines
**Alternative**: Generated automatically during build

## Testing & Coverage

### coverage/

**Why**: Test coverage reports (generated)
**Risk**: Large files, not needed in repository
**Alternative**: Generate locally with `npm test -- --coverage`

### test-results/, playwright-report/

**Why**: Playwright test results and reports
**Risk**: Large files, machine-specific
**Alternative**: Generate locally during testing

### .nyc_output/

**Why**: NYC coverage tool output
**Risk**: Temporary files
**Alternative**: Generated during test runs

## IDE & Editor Files

### .vscode/, .idea/

**Why**: IDE-specific settings (personal preferences)
**Risk**: Conflicts between team members' settings
**Alternative**: Share workspace settings via separate config

### _.swp, _.swo, \*.swn

**Why**: Vim swap files (temporary)
**Risk**: Not needed in repository
**Alternative**: Automatically created/deleted by Vim

### _.sublime-project, _.sublime-workspace

**Why**: Sublime Text project files (personal)
**Risk**: Personal preferences
**Alternative**: Each developer maintains their own

## Operating System Files

### .DS_Store (macOS)

**Why**: macOS folder metadata
**Risk**: Not needed on other platforms
**Alternative**: Automatically created by macOS

### Thumbs.db (Windows)

**Why**: Windows thumbnail cache
**Risk**: Not needed on other platforms
**Alternative**: Automatically created by Windows

### Desktop.ini (Windows)

**Why**: Windows folder customization
**Risk**: Not needed on other platforms
**Alternative**: Automatically created by Windows

## Logs & Debug Files

### \*.log

**Why**: Application logs (runtime data)
**Risk**: Can be very large, contains runtime info
**Alternative**: Configure logging to separate directory

### logs/

**Why**: Log directory
**Risk**: Large files, runtime-specific
**Alternative**: Configure log rotation and archival

## Temporary Files

### _.tmp, _.temp, .cache/

**Why**: Temporary files created during execution
**Risk**: Not needed in repository
**Alternative**: Automatically created/deleted

### tmp/, .temp/

**Why**: Temporary directories
**Risk**: Runtime-specific files
**Alternative**: Created as needed

## Package Manager Files

### package-lock.json (optional)

**Why**: Locks exact dependency versions
**Note**: Currently ignored, but you may want to commit it for reproducible builds
**Alternative**: Commit for production projects, ignore for libraries

### yarn.lock, pnpm-lock.yaml

**Why**: Alternative package manager lock files
**Note**: Only relevant if using Yarn or pnpm
**Alternative**: Commit the lock file for your package manager

## Playwright Specific

### .playwright/

**Why**: Playwright browser binaries cache
**Risk**: Very large (100MB+ per browser)
**Alternative**: Install with `npx playwright install`

### playwright/.cache/

**Why**: Playwright cache directory
**Risk**: Large, machine-specific
**Alternative**: Automatically managed by Playwright

## Backup Files

### _.bak, _.backup, \*~

**Why**: Backup files created by editors
**Risk**: Not needed in version control
**Alternative**: Use Git for versioning

## Best Practices

### 1. Never Commit Credentials

- Always use `.env` files or config files
- Add them to `.gitignore` immediately
- Use `config.example.json` as template
- Document required variables

### 2. Keep Repository Clean

- Ignore build artifacts
- Ignore IDE-specific files
- Ignore OS-specific files
- Ignore temporary files

### 3. Document Ignored Files

- Explain why files are ignored
- Provide alternatives
- Update `.gitignore` as project evolves

### 4. Test Before Committing

```bash
# Check what will be committed
git status

# Check if sensitive files are ignored
git check-ignore -v config/config.json

# Review changes before committing
git diff --cached
```

### 5. Use Global .gitignore

Create `~/.gitignore_global` for personal preferences:

```bash
# Configure global gitignore
git config --global core.excludesfile ~/.gitignore_global

# Add personal preferences
echo ".DS_Store" >> ~/.gitignore_global
echo ".vscode/" >> ~/.gitignore_global
```

## Verification Checklist

Before committing, verify:

- [ ] `config/config.json` is NOT in git status
- [ ] `node_modules/` is NOT in git status
- [ ] `.env` files are NOT in git status
- [ ] `dist/` or `build/` is NOT in git status
- [ ] No `.log` files in git status
- [ ] No IDE files (`.vscode/`, `.idea/`) in git status
- [ ] No OS files (`.DS_Store`, `Thumbs.db`) in git status

## Emergency: Accidentally Committed Secrets

If you accidentally committed sensitive files:

### 1. Remove from Git History

```bash
# Remove file from Git but keep locally
git rm --cached config/config.json

# Commit the removal
git commit -m "Remove sensitive config file"

# Push changes
git push
```

### 2. Rotate Credentials

- Change all passwords immediately
- Regenerate API keys
- Update proxy credentials
- Notify team members

### 3. Clean Git History (if already pushed)

```bash
# Use BFG Repo-Cleaner or git filter-branch
# WARNING: This rewrites history!

# Install BFG
# Download from: https://rtyley.github.io/bfg-repo-cleaner/

# Remove file from all commits
java -jar bfg.jar --delete-files config.json

# Clean up
git reflog expire --expire=now --all
git gc --prune=now --aggressive

# Force push (coordinate with team!)
git push --force
```

### 4. GitHub/GitLab Specific

- If pushed to GitHub: Consider repository as compromised
- Rotate all credentials
- Consider making repository private
- Review GitHub's guide: https://docs.github.com/en/authentication/keeping-your-account-and-data-secure/removing-sensitive-data-from-a-repository
